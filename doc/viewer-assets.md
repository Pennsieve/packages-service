# Viewer Assets

Viewer assets are auxiliary files associated with packages in a dataset that power interactive visualizations in the Pennsieve web application. They are distinct from the source files a user uploads — they are derived artifacts (e.g., OME-Zarr pyramids, Neuroglancer precomputed volumes, Parquet UMAP plots) generated by compute workflows and stored alongside the original data.

## Data Model

### Relationships

```
Dataset
 └── Package (source file)
       └── Viewer Asset (1 or more per package)
             └── S3 objects (the actual files)
```

- A **viewer asset** belongs to exactly one **dataset**.
- A viewer asset can be linked to **one or more packages** (many-to-many via a junction table).
- Each viewer asset maps to a set of files stored under a single S3 prefix.

### Database Tables

Viewer assets are stored in the **organization-scoped PostgreSQL schema** (`"{org_id}"`):

**`viewer_assets`**

| Column       | Type      | Description                                      |
|--------------|-----------|--------------------------------------------------|
| `id`         | UUID      | Primary key                                      |
| `dataset_id` | int       | Foreign key to `datasets`                        |
| `name`       | string    | Human-readable name (e.g., `"umap-projection"`)  |
| `asset_type` | string    | Type identifier (e.g., `"ome-zarr"`, `"parquet-umap-viewer"`) |
| `properties` | JSONB     | Arbitrary metadata for the viewer                |
| `s3_bucket`  | string    | Bucket where files are stored                    |
| `status`     | string    | Lifecycle state: `created` or `ready`            |
| `created_by` | int       | User ID                                          |
| `created_at` | timestamp |                                                  |

**`viewer_asset_packages`** (junction table)

| Column            | Type | Description                        |
|-------------------|------|------------------------------------|
| `viewer_asset_id` | UUID | Foreign key to `viewer_assets.id`  |
| `package_id`      | int  | Foreign key to `packages.id`       |

### S3 Storage Layout

All viewer asset files live under a deterministic prefix:

```
s3://{bucket}/viewer-assets/O{OrgID}/D{DatasetID}/{AssetID}/
```

Example:
```
s3://pennsieve-storage/viewer-assets/O19/D2049/a1b2c3d4-e5f6-7890-abcd-ef1234567890/
  ├── data.parquet
  ├── metadata.json
  └── config.yml
```

### Asset Status Lifecycle

```
created  ──(files uploaded)──>  ready  ──(deletion)──>  cleanup queue  ──(async)──>  S3 objects removed
```

- **`created`**: Record exists, upload in progress. Not yet visible to viewers.
- **`ready`**: All files uploaded. Available to the front-end.
- On **deletion**, a database trigger queues the S3 prefix for async cleanup by a separate Lambda.

## End-to-End Workflow: Creating Viewer Assets

The typical flow is driven by a **Step Functions workflow** that orchestrates compute processors. The final stage invokes the `data-target-assets` processor, which creates the viewer asset record and uploads the generated files.

```
 ┌──────────────────────────────────────┐
 │  Step Functions Orchestrator         │
 │  (invokes compute processors)       │
 └──────────────┬───────────────────────┘
                │
                ▼
 ┌──────────────────────────────────────┐
 │  Upstream Processor                  │
 │  (e.g., generates UMAP parquet      │
 │   files and writes to INPUT_DIR)    │
 └──────────────┬───────────────────────┘
                │
                ▼
 ┌──────────────────────────────────────────────────────────────────┐
 │  data-target-assets processor                                    │
 │                                                                  │
 │  1. Discover files in INPUT_DIR                                 │
 │  2. Resolve package IDs from the execution run                  │
 │  3. POST /packages/assets  →  creates record + gets credentials │
 │  4. Upload files to S3 using temporary STS credentials          │
 │  5. PATCH /packages/assets/{id}  →  status = "ready"           │
 └──────────────────────────────────────────────────────────────────┘
```

### Step-by-Step Detail

#### 1. File Discovery

The processor recursively walks `INPUT_DIR`, following symlinks, and collects all files to upload. No transformation occurs at this stage — the upstream processor has already produced the final asset files.

#### 2. Package ID Resolution

The processor determines which packages the asset should be linked to:

- If `PACKAGE_ID` is explicitly set, that single package is used.
- Otherwise, it fetches the execution run from the workflow service (`GET /compute/workflows/runs/{runId}`) and extracts package IDs from the run's data sources.

#### 3. Create Asset Record (POST /packages/assets)

The processor calls the packages-service API to create the viewer asset:

**Request:**
```json
{
  "name": "umap-projection",
  "asset_type": "parquet-umap-viewer",
  "properties": { "dimensions": 2, "metric": "cosine" },
  "package_ids": ["N:package:abc-123"]
}
```

**Response:**
```json
{
  "asset": {
    "id": "a1b2c3d4-...",
    "name": "umap-projection",
    "asset_type": "parquet-umap-viewer",
    "status": "created",
    "s3_bucket": "pennsieve-storage"
  },
  "upload_credentials": {
    "access_key_id": "ASIA...",
    "secret_access_key": "...",
    "session_token": "...",
    "expiration": "2026-04-10T15:00:00Z",
    "bucket": "pennsieve-storage",
    "region": "us-east-1",
    "key_prefix": "viewer-assets/O19/D2049/a1b2c3d4-.../"
  }
}
```

The response includes **temporary STS credentials** scoped to the asset's S3 prefix. These credentials only allow `PutObject` and multipart upload operations and expire after 1 hour.

#### 4. Upload Files to S3

Using the temporary credentials, the processor uploads each file via multipart upload (64 MB parts, SHA-256 checksums). The file's relative path within `INPUT_DIR` is preserved under the S3 key prefix.

#### 5. Mark Asset as Ready (PATCH /packages/assets/{id})

After all files are uploaded, the processor sets the asset status to `ready`:

```json
{ "status": "ready" }
```

The asset is now available for front-end viewers to consume.

## Packages-Service API Reference

### Authenticated Endpoints

All authenticated endpoints require `token_dataset_auth` (JWT via API Gateway authorizer).

#### POST /packages/assets

Create a viewer asset and receive temporary upload credentials.

- **Permission**: `EditFiles`
- **Query params**: `dataset_id` (required)
- **Returns**: 201 with asset record and `upload_credentials`

#### GET /packages/assets

List viewer assets linked to a package.

- **Permission**: `ViewFiles`
- **Query params**: `dataset_id` (required), `package_id` (required)
- **Returns**: Array of assets (each with `asset_url`) plus CloudFront signing params

#### PATCH /packages/assets/{asset_id}

Update an asset's status, properties, or linked packages.

- **Permission**: `EditFiles`
- **Query params**: `dataset_id` (required)
- **Body fields** (all optional): `status`, `properties`, `package_ids`

#### DELETE /packages/assets/{asset_id}

Delete an asset record and queue its S3 objects for cleanup.

- **Permission**: `EditFiles`
- **Query params**: `dataset_id` (required)
- **Returns**: 204

### Public Endpoint (Discover)

#### GET /packages/discover/assets

List viewer assets for a published package. No authentication required.

- **Query params**: `package_id` (required) — source package node ID
- **Validation**: Checks the Discover database to confirm the package is published
- **Returns**: Array of assets (each with `asset_url`) plus CloudFront signing params

## Accessing Viewer Assets from the Front-End

1. **List assets** for a package using `GET /packages/assets` (authenticated) or `GET /packages/discover/assets` (public).
2. Each asset in the response includes an `asset_url` — the full CloudFront base URL for that asset's files.
3. The response also includes `cloudfront` signing params (`policy`, `signature`, `key_pair_id`).
4. Construct file URLs by appending the relative file path to the asset's `asset_url` and adding the signing params as query parameters.
5. Signed policies expire after **1 hour**.

Example response:
```json
{
  "assets": [
    {
      "id": "a1b2c3d4-...",
      "name": "ome-zarr",
      "asset_type": "ome-zarr",
      "asset_url": "https://assets.pennsieve.net/abc123ef/O19/D2049/a1b2c3d4-.../",
      "status": "ready",
      "package_ids": ["N:package:..."]
    }
  ],
  "cloudfront": {
    "policy": "eyJ...",
    "signature": "gkv...",
    "key_pair_id": "K135..."
  }
}
```

Example file URL construction:
```
{asset_url}{relative_path}?Policy={policy}&Signature={signature}&Key-Pair-Id={key_pair_id}

https://assets.pennsieve.net/abc123ef/O19/D2049/a1b2c3d4-.../.zattrs?Policy=eyJ...&Signature=gkv...&Key-Pair-Id=K135...
```

## Authentication

| Context | Mechanism |
|---------|-----------|
| Front-end / user API calls | JWT token via API Gateway authorizer (dataset-scoped) |
| data-target-assets processor | Callback token: `Authorization: Callback workflow-service:{runId}:{token}` |
| S3 upload | Temporary STS credentials (1-hour TTL, scoped to asset prefix) |
| S3 read (front-end) | CloudFront signed policy (1-hour TTL) |

## Asset Cleanup

A scheduled Lambda (`packages-service-asset-cleanup`) runs every hour. It:

1. Polls the `viewer_asset_cleanup_queue` table (up to 100 entries per run).
2. Deletes all S3 objects under each queued prefix.
3. Removes the queue entry after successful deletion.

This ensures S3 storage is reclaimed even if inline deletion during the `DELETE` API call fails or times out.
